# Agentic Chat Loop

*Status:* Currently `planner.py` generates a static JSON plan from a prompt. You want a conversation.
*Goal:* A Read-Eval-Print Loop (REPL) where you chat, and the AI executes tools immediately.

## Initial thoughts

1.  **Refactor `planner.py`:** Instead of generating a full list `[...]`, switch the system prompt to "ReAct" (Reasoning + Acting) style.
2.  **The Loop (Host Side):**
    *   User types "Dispense 100ul into A1".
    *   LLM sees tools (`sidekick`, `colorimeter`).
    *   LLM output: `{"tool": "sidekick", "cmd": "dispense_at", "args": {...}}`.
    *   Host executes immediately via `DeviceManager`.
    *   Host feeds result (`SUCCESS` or `DATA_RESPONSE`) back to LLM context.
    *   LLM replies to user: "Done. What next?"
3.  **Integration:** Create `host/ai/ai_chat.py` which unifies `ai_plan.py` and `ai_execute.py` into this single loop.
4.  **Remove LLM ties:** `host/ai/vertext_agent.py` is currently tied to a single LLM provider which makes it difficult to use other such as Ollama for local serving of the LLM. It might be better to implement a "Postman Pattern" similar to what is in `communicate/` to resovle.
    *  Create a `BaseAgent` abstract class
    *  Implement `VertextAgent` and `OllamaAgent` subclasses
    *  The `Agent` class should just be a wrapper that takes `params` dictionary e.g. `{"provider": "ollama", "model": "llama3"}` to decide which backend to use. Implementing the OpenAI API format and targeting OpenAI-compatible will simplify this part of the project.
5.  **MCP Integration:** Implementing a standard that allows the AI to discover tools. MCP will likely make the current task - an agentic chat loop - obsolete; however the MCP server would require significant overhead to get the features operational. We want to address the chat loop first using existing infrastructure and start **standardizing tool definitions** with clean descriptions and args schema such as those in `sidekick/__init__.py`.

## Possible roadmap

### Stage 1: Abstract the LLM Postman

*  Refactor `vertext_agent.py` into a provider-agnostic `llm_manager.py`.
*  Utilize a strategy similar to the Postman Pattern in `communicate` and generate a `BaseAgent` abstract class with `VertextAgent` and `OllamaAgent` subclasses.
    *  Avoid the use of ABC and abstractmethod in the abc library, which is not part of circuitpython and therefore incompatible with the hardware.
    *  Implement OpenAI API format where possible to target OpenAI compatible solutions.
*  **Success condition:** Can swap between Gemini and a local Ollama instance. 

### Stage 2: The "Stateful Tool user" (Logic)

*  Refactor `planner.py` logic to handle single-turn execution
*  Modify `PlateManager` to be able to "undo" or "verify" state if a hardware command fails (received a `PROBLEM` message).
*   **Success Condition:** The system can successfully execute a **conditional multi-step task** where the second step is generated only after the first step's result is processed.
    *   *Test Scenario:* A script is given the prompt: *"Add 100µL to well A1. If and only if that succeeds, measure the spectrum."*
    *   *Validation:* The logs must show the `sidekick` command executing, the `DeviceManager` receiving a `SUCCESS` message, the `PlateManager` updating the volume of A1, and *then* the LLM receiving that confirmation as context before it generates the `colorimeter` command.

### Stage 3: The "REPL" (Interaction)

*  Create `host/ai/ai_chat.py`
*  The Loop:
    *  User Input ->
    *  LLM Reasoning ->
    *  Tool Call Proposal ->
    *  Human Confirmation ->
    *  `DeviceManager` execution ->
    *  Parse `DATA_RESPONSE` ->
    *  LLM summarizes result to User.
*   **Success Condition:** A continuous, conversational session where the AI maintains **contextual memory** across multiple human interventions and device responses.
    *   *Test Scenario:* 
        1. User: *"Move the arm to A1."* (AI moves arm).
        2. User: *"What's the current position?"* (AI calculates world coordinates via `get_info` and answers).
        3. User: *"Dispense 50µL of red reagent here."* (AI identifies "here" as A1 and identifies "red" as Pump 1 from the World Model, then executes).
    *   *Validation:* The AI must correctly resolve "here" and "red" based on the ongoing conversation and the `PlateManager` state without the user repeating the coordinates or pump IDs.

## Next project: Hardware-as-a-Service (MCP)

Upon completing and validating the agentic chat loop, we will move on to standardizing the lab for the wider AI ecosystem. This branch will need additional planning, but the initial ideas are the following

*  Wrap the `DeviceManager` in an MCP server. This allows the lab to be "plugged in" to any modern agentic framework without writing any more custom chat code.
*   **Success Condition:** An **external, off-the-shelf MCP client** (e.g., Claude Desktop, `mcp-inspector`, or an IDE-based agent) can automatically discover and control the laboratory instruments without any custom ALIF-specific integration code.
    *   *Test Scenario:* Point an MCP-compliant client at the ALIF MCP server.
    *   *Validation:* The client automatically populates its "Toolbox" with entries like `sidekick_move_to` and `colorimeter_measure` (derived from your `handlers.py` documentation), and a command triggered from the external client successfully moves physical hardware.
