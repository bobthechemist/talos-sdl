# Project Requirements Document
**Project:** ALIF Agentic Laboratory Cockpit – Refactored Chat Loop  
**Version:** 1.0 (Final)  


---

## 1. Executive Summary
Refactor the chat loop to:
- Support firmware‑resident deterministic **skills** (e.g., `optimize_gain`) that encapsulate closed‑loop calibration/optimization on the instrument.
- Introduce explicit, user‑controlled modes: **/run** (instrument control; strict JSON plans) and **/data** (analysis; no device I/O).
- Use a **single LLM agent** with **mode‑specific contexts** and **separate histories**, plus a **temporary one‑shot analysis** in /run that summarizes `DATA_RESPONSE` without changing modes.

Design must anticipate future features (multi‑agent routing, control DSL, reflexive planning) without increasing complexity for v1.

---

## 2. Goals & Objectives
- **G1:** Deterministic, firmware‑driven procedures callable through a stable JSON planning interface.
- **G2:** Explicit, intentional mode switching via slash commands to prevent accidental actuation.
- **G3:** Single agent, multiple contexts (RUN/DATA), isolated histories; temporary, no‑history **one‑shot analysis** in /run.
- **G4:** Preserve the existing JSON plan schema; **no host‑side DSL** in v1.
- **G5:** Strong logging/observability, dataset registry, and artifact lineage.
- **G6:** Architecture ready for future **multi‑agent routing** and **control DSL**.

---

## 3. In‑Scope (v1)
- Firmware **skills** integration (host invokes skills as normal commands; instrument performs closed‑loop logic).
- **/run** and **/data** modes with explicit user control.
- Single agent with mode‑specific system contexts and histories.
- **Plan review** gate in /run (confirm/edit/reject).
- **One‑shot analysis** in /run upon `DATA_RESPONSE` (no mode switch).
- Minimal **analysis tools** in /data (list/summarize/plot).
- **Dataset registry** and **session logging**.
- Safety: timeouts, validation, dry‑run.

---

## 4. Out‑of‑Scope (v1)
- Host‑side **control DSL** (loops/ifs).
- **Reflexive** multi‑turn planning with observation between each step.
- On‑device skill authoring from host.
- Advanced analytics beyond summarize/plot.
- Cloud/remote multi‑user services.

---

## 5. Stakeholders & Roles
- **Product/PI:** Scientific goals, acceptance criteria.
- **Firmware Team:** FSM skills, capability schema, deterministic routines.
- **Host Team:** REPL, Planner/Executor, modes, logging, analysis tools.
- **Operators/Researchers:** Use /run and /data; provide usability and safety feedback.

---

## 6. Key Assumptions
- Firmware exposes **self‑documenting capabilities/skills** via structured help (`DATA_RESPONSE`) including args, constraints, and return schema.
- Instruments are deterministic; closed‑loop logic resides on the device.
- DeviceManager and message protocol remain stable.
- Single LLM agent suffices for v1; context isolation is feasible.

---

## 7. Functional Requirements

### 7.1 Mode System (REPL & Routing)
- **FR‑RUN‑1:** `/run` activates instrument‑control mode; agent responses must be **JSON plans only**.
- **FR‑RUN‑2:** `/run` enforces **plan review**: `run/y`, `reject/n`, `del <#>`, `edit <#>`.
- **FR‑RUN‑3:** `/run` prohibits analysis tools except one‑shot summaries (see 7.4.1).
- **FR‑DATA‑1:** `/data` activates analysis mode; **no device I/O** permitted.
- **FR‑DATA‑2:** `/data` allows natural‑language replies and use of analysis tools.
- **FR‑MODE‑UX:** Provide `/mode`, `/status`, `/help` with clear state feedback.

### 7.2 Firmware Skills (Closed‑Loop Procedures)
- **FR‑SKILL‑1:** Host enumerates device **skills** from the capability/help payload at startup.
- **FR‑SKILL‑2:** Skills are invoked as normal commands in plans:
    - Example:
        - { "device": "colorimeter", "command": "optimize_gain", "args": { "channel": "yellow", "target_pct": 98, "backoff_levels": 1 } }
- **FR‑SKILL‑3:** Skills return **structured** `DATA_RESPONSE` results (e.g., `final_gain`, `iterations`, `trace`).
- **FR‑SKILL‑4:** Host validates args against the device capability schema; rejects invalid plans at review.

### 7.3 Planner / Agent Behavior
- **FR‑PLAN‑1:** Planner RUN context lists available commands/skills, operational guidance, world model, and strict JSON rules.
- **FR‑PLAN‑2:** Planner leverages world model safety constraints (e.g., volume caps).
- **FR‑AGT‑1:** Single agent with **two contexts** and **two isolated histories** (/run and /data).
- **FR‑AGT‑2:** Switching modes must **not** leak history; the system switches system prompts and uses the appropriate history.

### 7.4 Data Analysis

#### 7.4.1 One‑Shot Analysis in `/run` (NEW)
- **FR‑RUN‑ANALYSIS‑1:** When a `DATA_RESPONSE` is received in /run, the system invokes a **temporary analysis context** using the **same agent**, with:
  - Short, deterministic system instruction.
  - **No device‑control affordances**.
  - **No history** (does not append to /run or /data histories).
  - Input template:
        - The user asked: "<user_prompt>"
          I executed: <device>.<command>(<args>)
          I received: <payload>
          Summarize concisely and answer the user’s question.
- **FR‑RUN‑ANALYSIS‑2:** Output a **human‑readable summary** (e.g., “LED intensity is 4 mA; LED is off.”) and remain in /run.
- **FR‑RUN‑ANALYSIS‑3:** The temporary analysis must never emit device commands or plan JSON.
- **FR‑RUN‑ANALYSIS‑4:** Print a hint: “Type `/data` for deeper analysis or `/explain last` to revisit this summary.”
- **FR‑RUN‑ANALYSIS‑5:** Store the raw `DATA_RESPONSE` and the **generated summary** for recall.

#### 7.4.2 Minimal Analysis Tools in `/data`
- **FR‑DATA‑TOOLS:** Provide tools:
  - `list_datasets()`
  - `describe(id)`
  - `summarize(id, columns)`
  - `plot(id, x, y, kind)`
- **FR‑DATA‑EXEC:** Run tools in a constrained sandbox; save artifacts to `temp/analysis` and `temp/plots`.
- **FR‑DATA‑NOACT:** `/data` forbids any device I/O.

### 7.5 Dataset Registry & Artifacts
- **FR‑DS‑1:** Register datasets on each `DATA_RESPONSE` (and analysis outputs) with:
  - `id`, `path`, `schema`, `origin_device`, `origin_command`, `timestamp`, `tags`.
- **FR‑DS‑2:** Provide `/datasets` and `/artifacts` commands to browse.

### 7.6 Logging & Observability
- **FR‑LOG‑1:** Extend `session_history` with: session start, mode switches, plans, confirmations, step results, datasets, artifacts, errors.
- **FR‑LOG‑2:** `executor.save_log()` writes JSON to `temp/chat-session-*.json`.
- **FR‑LOG‑3:** For skills, capture and store iteration traces returned by firmware.
- **FR‑LOG‑4:** Log one‑shot summaries tagged as `"analysis_reply": "one_shot"` and link to the originating `DATA_RESPONSE`.
- **FR‑LOG‑5:** Store raw `DATA_RESPONSE` for `/last` and `/explain last`.

### 7.7 Safety & Guardrails
- **FR‑SAFE‑1:** Timeouts and retries on device comms; safe aborts.
- **FR‑SAFE‑2:** Strict validation: unknown device/command/args → reject at review.
- **FR‑SAFE‑3:** `/data` enforces **no device actions** at routing and execution layers.
- **FR‑SAFE‑4:** Default **dry‑run** option in /run (configurable).
- **FR‑SAFE‑5:** Temporary analysis in /run is sandboxed to **prohibit device commands**.
- **FR‑SAFE‑6:** If the question cannot be answered from the payload, the summary must say so and suggest `/data`.

### 7.8 Error Handling
- **FR‑ERR‑1:** Non‑JSON LLM output in /run → treated as completion/refusal; no actuation.
- **FR‑ERR‑2:** Device errors → clear message, abort safely.
- **FR‑ERR‑3:** `/data` tool errors return structured messages without terminating the session.

### 7.9 Performance
- **FR‑PERF‑1:** Incremental logging to avoid data loss.

### 7.10 UX Commands (REPL)
- `/run`, `/data`, `/mode`, `/status`, `/help`, `/clear`, `/log`, `/datasets`, `/artifacts`, `/dryrun on|off`
- `/explain last` (replay last one‑shot summary; no I/O), `/last` (print raw last `DATA_RESPONSE`), `/summary` (recap: last command, last dataset, last summary).

## 10. Architecture Overview (v1)
- **REPL (`chat.py`)**: Parses slash commands; manages current mode; displays status/help. Delegates to **AgentExecutor** for /run and **Analysis tools** for /data. Provides `/explain last`, `/last`, `/summary`.
- **Planner**: Builds mode‑specific system prompts. RUN prompt enumerates commands/skills and strict JSON response rules.
- **Agent (single)**: One instance, two contexts, two histories. Temporary **one‑shot analysis** in /run uses the same agent with **no history** and **no device tools**.
- **AgentExecutor**: Validates plans; plan review; sends commands; waits for results; updates logs, plate state, and dataset registry; triggers one‑shot analysis on `DATA_RESPONSE`.
- **Analysis Tools**: Minimal local toolkit for /data; no device access.
- **DeviceManager**: Unchanged serial/message I/O.
- **Firmware**: FSM with self‑documented skills; deterministic routines.

---

## 11. Acceptance Criteria

### /Run Mode
- Agent outputs **JSON‑only**; non‑JSON handled gracefully (no actuation).
- Human can **review/edit/reject** plans.
- Skill invocation (e.g., `optimize_gain`) completes with structured results logged.
- On `DATA_RESPONSE`, a **natural‑language** one‑shot summary is printed; mode remains /run.

### /Data Mode
- No device commands are issued.
- `list_datasets`, `summarize`, `plot` operate on sample datasets; artifacts saved and listed.

### General
- `/mode`, `/status`, `/help` reflect state.
- `save_log` includes plans, steps, datasets, artifacts, one‑shot summaries, errors.

---

## 12. Risks & Mitigations
- **R1: Firmware bottleneck for new skills.**  
  *Mitigation:* Clear skill schema; prioritize a core set; version skills.
- **R2: Context leakage between modes.**  
  *Mitigation:* Separate histories; hard prompt switch; no‑history temporary analyses.
- **R3: LLM fabricates commands.**  
  *Mitigation:* Validate strictly against discovered capability schema; review gate.
- **R4: User confusion about modes.**  
  *Mitigation:* Prominent status; `/mode` echo; `/help`; hints after one‑shot summaries.
- **R5: Analysis scope creep.**  
  *Mitigation:* Minimal toolset in v1; stable extensible interfaces.
