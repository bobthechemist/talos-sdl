# future directions

I have settled on **talos-sdl** as the official name of the project. I have also been spending some time thinking about what version 2.0 will look like. I provided this *vision* to a chatbot to see how far away we are.

>So let's say that in version 2.0, I would like to move away from dependence on the agentic chat loop and explore a more autonomous agent. In the human would connect sensors (sidekick 2: a newer version that has a tool changer with some new sensors) and establish the world conditions (empty well plates, a metal ion and two different chelating agents in the pump reservoirs) and then the human only issues the command /wakeup at which point the agent recognizes it has not been told to do something, so it looks at its world: it sees the information from Platemanager and the world file; it finds out what tools it has available; and it checks an electronic lab notebook (what did I do yesterday?) It may have some prior content knowledge (some papers on coordination chemistry, Job plots), and it notices that previous work was done with the same metal it has access to, but the two chelating ligands are new. At this point, the agent should be able to craft a plan to explore some chemistry along the trajectory of prior work but with the world it has been given today. The agent then decides on a project (explore coordination chemistry with one of the chelating compounds, or is interested in making mixed-ligand complexes) and starts performing some chemistry. It knows (because of the world) that it can do 96 experiments (one in each well) and then must wrap up work for the day and write a report. That report gets stored in memory and is reviewed by the human, who resents the agent's world with a fresh well plate, new or refreshed chemicals, and a new day repeats. How far is Talos-SDL away from this reality?


## Talos-sdl  2.0

Here is a summary of what it might take for *Talos-sdl V2.0* to be realized.

### Project Snapshot & v2.0 Vision Brief: Talos-SDL
**Target Project:** Talos-SDL (Agentic Self-Driving Laboratory Orchestrator)
**Target Version:** 2.0 (The "Autonomous Scientist" Update)

#### 1. Project Background & Current State (v1.0)
Talos-SDL is an open-source software framework that orchestrates physical laboratory hardware to perform automated chemistry and materials science experiments. 

Currently, v1.0 successfully operates as an **Instruction-Following Agent** (a robotic technician). The hardest physical and software engineering challenges have already been solved:
*   **Infrastructure:** We use a strict Host-Device split. The Python host runs a Model-View-Controller (MVC) architecture, while physical microcontrollers run deterministic, non-blocking Finite State Machine (FSM) firmware. 
*   **Communication:** Standardized JSON over serial (`INSTRUCTION`, `DATA_RESPONSE`, `SUCCESS`).
*   **State Management:** The host accurately tracks physical state via a `PlateManager` (tracking liquid volumes and contents across a 96-well plate) and a `world_model.json` (tracking available reagents and physical limits).
*   **Current AI Loop:** A single LLM operates in an "Agentic REPL." A human types a natural-language goal, the AI generates a JSON execution plan based on dynamically discovered hardware capabilities, the hardware executes it, and the AI performs a one-shot analysis of the resulting data. It then waits for the next human prompt.

#### 2. The Vision for Version 2.0
The objective of v2.0 is to evolve Talos-SDL from a human-prompted "Technician" into a goal-generating **"Autonomous Scientist."** 

We want to move away from the turn-by-turn chat loop. In v2.0, the human operator's role shifts to lab preparation and data review. The human prepares the physical world (loads an empty 96-well plate, restocks pump reservoirs with a metal ion and chelating agents, updates the `world_model`), and issues a single command: `/wakeup`.

Upon waking, the system must autonomously:
1.  **Contextualize:** Read its current physical constraints (available reagents, 96 empty wells).
2.  **Recall:** Access an Electronic Lab Notebook (ELN) to review past experiments ("What did I do yesterday?") and consult a local literature repository (RAG) regarding the specific chemistry it has access to.
3.  **Hypothesize & Plan:** Autonomously generate a scientific objective for the day (e.g., "Yesterday I explored Ligand A; today I will run a Job plot for Ligand B to determine stoichiometry").
4.  **Execute & Analyze:** Enter an autonomous "outer loop" where it conducts experiments, analyzes the spectral data, and adjusts its subsequent experiments based on those findings.
5.  **Conclude:** Recognize when physical resources are exhausted (e.g., the 96-well plate is full), safely park the hardware, generate a comprehensive daily markdown report, save it to the ELN, and shut down to await the human's physical reset the next morning.

#### 3. Required Architectural Shifts for v2.0
To support this vision, the PRD should focus on building a new **Cognitive Architecture** on top of the existing physical execution layer. This requires:
*   **Multi-Agent Hierarchy:** Splitting the single agent into specialized personas:
    *   *The PI (Principal Investigator) Agent:* Handles long-term memory, hypothesis generation, and daily campaign planning.
    *   *The Technician Agent:* Translates the PI's goals into the specific JSON hardware plans (our current `Planner`).
    *   *The Analyst Agent:* Processes incoming datasets, updates the ELN, and feeds conclusions back to the PI.
*   **Long-Term Memory:** Implementing episodic memory (reading/writing to the ELN archive) and semantic memory (a RAG system for querying scientific papers/literature).
*   **The Autonomous Outer Loop:** A programmatic loop that replaces the REPL, allowing the agents to pass context to one another continuously until a physical stop-condition (like plate capacity) is met.

*Note to PRD Agent: Do not rewrite or alter the `DeviceManager`, FSM device firmware, or JSON communication protocols. The focus must be entirely on the Python-side autonomous orchestration, RAG integration, and multi-agent state loops.*

### Version 1 reality 

Below is a critical assessment of where things stand for version 1.0 and where they need to get to for 2.0

#### 1. World State, Tools, & Hardware (Very Close)
Your current framework handles this beautifully. 
*   **What you have:** The `DeviceManager` dynamically queries devices via the `help` command to discover capabilities. The `world_model.json` sets up the reagents. The `PlateManager` tracks the 96 wells.
*   **What is missing:** 
    *   *Tool Changer Logic:* This is just a new firmware handler (e.g., `swap_tool`) and an update to the physical Sidekick. Talos will handle this natively.
    *   *Resource Awareness:* The AI needs a programmatic prompt injection that says: `"You have X empty wells remaining. If X < 5, you must initiate the shutdown and reporting sequence."`

#### 2. The Electronic Lab Notebook & Past Memory (Moderate Gap)
Currently, Talos has **write-only long-term memory**. It generates excellent JSON logs and datasets via the `Registry`, but the agent does not read them when it boots up.
*   **What you have:** `AgentExecutor.save_log()` and `Registry.register()`.
*   **What is missing:** A **Memory Retrieval System**. When the human types `/wakeup`, the system needs to run a script that pulls the last `experiment_report.md` or the previous day's JSON summary and injects it into the LLM's system context: *"Here is what you accomplished yesterday..."*

#### 3. Literature & Prior Content Knowledge (Major Gap)
Currently, your agent relies entirely on the base weights of the LLM (e.g., Gemini 2.5 Flash) for its chemistry knowledge. 
*   **What you have:** Nothing explicit for literature retrieval.
*   **What is missing:** A **Retrieval-Augmented Generation (RAG)** module. You would need a folder (e.g., `docs/literature/`) containing PDFs or text files of papers on Job plots and coordination chemistry. A pre-processing script would vectorize these papers so the agent can query them to formulate scientifically valid hypotheses based on the specific chelating ligands in the `world_model`.

#### 4. Goal Generation: The "Outer Loop" (Major Gap)
Currently, your `chat.py` loop waits for a human to type a prompt. The agent solves *that specific prompt* and stops. 
*   **What you have:** The ReAct execution loop inside `AgentExecutor.run()`.
*   **What is missing:** The **Autonomous Outer Loop**. You need to wrap the `AgentExecutor` in a higher-level loop that operates without human input until a stop condition (like running out of wells) is met.

### The Blueprint for Talos-SDL V2.0

To achieve this reality, you need to transition from a single LLM to a **Multi-Agent Hierarchy**. You need to split the "Brain" into three distinct personas. 

Here is how you would structure the V2.0 Python architecture:

#### Agent 1: The Principal Investigator (PI)
*   **Role:** Hypothesis generation and campaign planning.
*   **Trigger:** The `/wakeup` command.
*   **Action:** It reads the `world_model.json`, reads the ELN from yesterday, and optionally searches a local literature database. 
*   **Output:** It generates a high-level scientific directive. *(e.g., "Today, let's determine the stoichiometry of the metal with Ligand A using a Job plot across 12 wells.")*

#### Agent 2: The Technician (Your current `Planner`)
*   **Role:** Translating high-level directives into hardware JSON plans.
*   **Trigger:** Receives the daily directive from the PI.
*   **Action:** Looks at the `PlateManager` to find the next 12 empty wells, calculates the required volumes, and generates the exact JSON `INSTRUCTION` payloads.
*   **Output:** Hardware actuates. Data is collected.

#### Agent 3: The Data Analyst (Your current `one_shot_analysis`)
*   **Role:** Interpreting results and updating the ELN.
*   **Trigger:** Receives the `DATA_RESPONSE` payloads from the Colorimeter.
*   **Action:** Analyzes the spectra. Finds the peak absorbance. Calculates the stoichiometric ratio.
*   **Output:** Writes a summary paragraph. Passes it back to the PI Agent.

#### The Loop
The PI reads the Analyst's summary. If the Plate is not full, the PI decides what to do next *(e.g., "The Job plot for Ligand A is complete and shows a 1:2 complex. Let's do Ligand B now.")*. It passes a new directive to the Technician. 

When the `PlateManager` reports 0 empty wells, the PI triggers `summarize.py`, writes the daily markdown report, saves it to the ELN, and safely parks the hardware.

### Conclusion

You are not far away. **You do not need to rewrite your firmware, your message protocol, or your DeviceManager.** 

To get to V2.0, you simply need to write a new Python script (e.g., `autonomous_campaign.py`) that instantiates three LLMs instead of one, feeds them specific prompts, and loops them together programmatically until the `PlateManager` is full. The hardest engineering challenges—getting an AI to safely and deterministically move a physical robot without breaking it—are already solved in your current codebase.

**Consider this** The current architecture can be described as a pull-based MVC where the view (UI) polls the model. The controller (DeviceManager) is checked by the View using some type of timer and if it finds data, it reaches to the model (Device) to grab what it needs to display. Fully agentic approaches will require a MVP switch where the controller becomes a presenter and handles all communication between view and model. If the decision is made to develop a MCP server, this change will be necessary.
